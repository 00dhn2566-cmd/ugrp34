## 미해결 — PID 재튜닝 (`tune_pid.m`)

FX450 CAD로 교체하면서 질량/관성모멘트가 바뀌었는데 PID 게인은 원래 MathWorks 예제 값 그대로라서, 지금 나오는 `motor_cmd_w1~w4` 값이 비정상적으로 큼 (평균 3만~9만대 rad/s, 정상 범위는 수백 rad/s 수준이어야 함, 음수 역회전 값도 나옴). **PID 재튜닝을 마쳐야 모터 명령 값을 신뢰할 수 있음.**

### 오늘 순서대로 잡은 에러들 (전부 실제로 고쳐짐)

1. **"has one or more input signal ports with no explicit line connections"**
   `Control Pitch` 안의 `PID Compensator Formula`는 실제 PID 블록이 아니라 **비활성 상태인 내부 구현 variant**(포트 0개)였음. → 마스크 블록 자체(`Control Pitch` 등)를 타겟으로 바꿔서 해결.
2. **이산(z=0) 적분기 zoh 변환 에러**
   `ST.Options.RateConversionOptions.Method = 'tustin';`로 해결.
3. **Tracking 목표 "9 vs 35" 크기 불일치**
   `From`(전체 9채널 레퍼런스 버스) vs `Quadcopter`(전체 35채널 상태 버스)를 통째로 io 포인트로 잡아서 발생. → `Scope/Demux`의 outport 1/2/3(des_x/y/z) vs `Scope/In Bus Element,1,2`(act_x/y/z, `Chassis.px/py/pz`) 스칼라 3쌍으로 교체해서 해결.
4. **"조정 가능한 파라미터가 없음"**
   `Control Pitch`의 P/I/D 필드(`attitude_kp` 등)는 사실 **부모 마스크** `Altitude and  YPR Control`이 노출하는 마스크 파라미터였음 (`attitude_kp = kp_attitude`, `yaw_kp = kp_yaw`, `altitude_kp = kp_altitude` 식으로, pitch/roll이 `attitude_*` 게인 공유). → 개별 `Control Pitch/Roll/Thrust/Yaw` 대신 부모 `Altitude and  YPR Control` 자체를 타겟으로 바꿔서 해결.
5. **"waypoints를 찾을 수 없음" 계열 연쇄 에러**
   `sim()`은 스크립트의 base workspace를 그대로 쓰지만, `slTuner`의 내부 배치 선형화(`compileForLinearization`)는 별도 컨텍스트라 base workspace의 `waypoints`/`wayp_path_vis`를 못 찾음. → 모델의 **Model Workspace**에 `mws.assignin(...)`으로 직접 넣어서 해결.

### 아직 안 풀린 것 — "InputName은 채널마다 하나의 이름을 지정해야 함"

- `Altitude and  YPR Control` 블록을 **단독으로** 튜닝 대상으로 넣어도 동일 에러 발생 → 이 블록 자체에 내재된 문제 (다른 블록과 합쳐서 생기는 이름 충돌이 아님).
- `Altitude and  YPR Control`의 입력 5개/출력 4개 포트는 전부 signal name이 빈 문자열("")로 확인됨.
- `ST.Options.UseFullBlockNameLabels = 'on'`으로도 해결 안 됨.
- **포트 이름을 실제로 유일하게 바꿔봐도(`YPR_` 접두어) 똑같은 에러** → 포트 이름 충돌 가설은 반증됨.
- `Position Control/PID Controller`를 단독으로 넣으면 "조정 가능한 파라미터 없음" 에러. 실제로 확인해보니 이것도 `Control Pitch`처럼 부모 마스크(`Position Control`, `kp_xy2ypr` 등)의 파라미터를 참조하는 구조였음 — 부모로 타겟 변경 시 이번엔 **"InputName" 에러가 아니라 다시 "조정 가능한 파라미터 없음"**이 남음 (원인 불명, `Position Control`과 `Altitude and YPR Control`의 `MaskTunableValues`는 둘 다 전부 `on`으로 확인되어 이 차이는 아님).
- `Simulink.Parameter`로 게인 변수를 감싸서 `slTuner(mdl, mdl, io)`(모델 전체 자동탐색)도 시도 → "튜닝할 블록을 지정해야 한다"고 거부됨. 이 방식 자체가 안 먹힘.

### 2차 세션: `systune`/`slTuner`를 포기하고 `linearize()` + `pidtune()` 직접 시도

`slTuner`의 `TunedBlocks` 메커니즘이 이 모델의 버스 기반 마스크 서브시스템 구조와 근본적으로 안 맞는 것으로 보여, 더 단순한 API로 우회 시도:

- `linearize(mdl, io, snapshotTime)`으로 `Position Control/PID Controller`의 출력→실제 위치(`act_x`) 구간을 직접 선형화 → **에러 없이 성공**하지만 결과가 `1×3` 크기의 **완전히 0인 시스템**(`A=[]`, `D=[0 0 0]`, `nstates=0`)으로 나옴. `pidtune`이 "SISO 플랜트가 아니다"로 거부.
- 스냅샷 시간을 5/10/20/30초로 바꿔봐도 **전부 동일하게 0** → 특정 시점 문제 아님.
- 더 단순한 내부 경로(`Position Control`의 `Traj` 입력 → `Pitch Cmd` 출력, 한 서브시스템 안)로 좁혀도 **여전히 정확히 0**.
- **포트 번호가 잘못됐나 확인** — `Position Control` 외부 Inport 1=`m`(Chassis 상태 버스), Inport 2=`Traj`(pos/vel/yaw 레퍼런스 버스), Outport 1=`Pitch Cmd`(내부적으로 `Pitch Limit` 블록에서 나옴) 맞게 짚은 것으로 확인됨 → 포트 번호 문제 아님.
- **Block Reduction 최적화가 io 지점을 최적화로 없애버리나** 의심 → `set_param(mdl,'BlockReduction','off')` 해도 **동일하게 0**.
- **`Pitch Limit`(Saturate, `[-pi/3, pi/3]`)가 포화돼서 그런가** 의심 → 실제 로깅해보니 `pitch_cmd`는 한계값에 안 붙고 `0.857`(범위 안)로 **안정적으로 수렴**해있음 → 포화 가설도 반증됨.
- 위치+자세 게인을 둘 다 훨씬 낮춰(`kp_position=0.5`, `kp_attitude=5`) 재시도해도 **여전히 0** (t=5/15/25 전부).
- **현재 남은 가설**: `PID Controller` 내부의 `Reset Signal`(외부 리셋) 로직이 시뮬레이션 내내 켜져 있어서 출력이 고정(freeze)돼 있고, 그래서 입력을 흔들어도(perturbation) 국소 미분값이 0으로 나오는 것 아닌가 — **확인 안 됨, 다음에 이어서 볼 것**.

> **⚠️ 6차 세션에서 정정됨**: 아래 3~5차 세션에 나오는 **"yaw" 관련 수치(425,135° / 369,439° / 365,507° 등)는 전부 잘못된 값**입니다. `Scope/In Bus Element11`을 "act_yaw"(Chassis.yaw)로 가정하고 탭했는데, 실제로는 **`Prop1.w`(모터1 각속도, rad/s)** 였습니다. 진짜 `Chassis.yaw`는 6차 세션에서 확인한 대로 시종일관 1° 미만이었고, yaw는 애초에 폭주한 적이 없습니다. roll/pitch(478.96° 등)는 실제 값이 맞습니다(별도 블록에서 탭했었음). 아래 3~5차 세션 내용은 "무엇을 잘못짚었는지"의 기록으로 남겨두고, 결론은 맨 아래 6차 세션을 참고하세요.

### 3차 세션: GUI PID Tuner + 실제 호버 테스트로 진짜 근본 원인 확정

GUI PID Tuner(`Control Pitch`)로 게인을 조절해봤을 때 스텝 응답 자체는 그럴듯해 보였지만, 나온 파라미터가 `P=-5.68e10, I=-5.97e11, D=2.19e9` 같은 천문학적 값이었음 → 실제로 물리 시뮬레이션에 적용하면 바로 발산할 값들. 이걸 계기로 궤적 없이 **"제자리에서 1m 높이로 5초간 가만히 떠있기"**라는 아주 단순한 호버 테스트를 만들어서 원래(수정 안 한) 게인으로 돌려봄:

- **z(고도)**: 1m → -0.014m (추락), **y**: 0 → 2.77m (드리프트)
- **roll**: 478.96°, **pitch**: 478.96°, **yaw**: **425,135°** (5초 만에 1180바퀴 이상 회전!)

즉 원래 게인은 **궤적 추종은커녕 제자리 호버링조차 못 함** — 명백한 발산. 특히 yaw가 압도적으로 심각해서 원인을 추적:

1. **가설 1: 프로펠러 회전방향(CW/CCW) 오류.** 4개 프로펠러 `Aerodynamic Propeller.direction`이 전부 `Positive`로 확인됨(정상 쿼드콥터는 대각선 2쌍이 반대 방향이어야 반작용 토크가 상쇄됨). Motor Mixer의 yaw 부호 그룹(모터1&4 vs 2&3, `Out Bus Element` 매핑 기준)에 맞춰 프로펠러 2,3을 `Negative`로 바꿔서 재검증:
   - **X/Y 드리프트는 거의 완전히 해결됨** (y: 2.77m → 0.0016m) — 실질적 개선.
   - 하지만 **yaw 값은 수정 전후로 소수점까지 완전히 동일**(425135 vs 425135) → `direction` 파라미터가 애초에 yaw 반작용 토크 계산에 반영이 안 되는 구조로 보임 (추력 부호에만 영향). 이 가설은 yaw에 대해서는 반증됨.

2. **가설 2: 모터 전기 속도 서보(`Control1~4`, `kp_motor=0.00375`)가 너무 약해서 차등 속도 제어가 안 걸림.** 모터 1~4의 `ref`(명령)/`meas`(실측 속도)를 직접 로깅 → **4개 모터 전부 정확히 같은 속도(7420.01 rad/s)로 수렴**, 각자 다른 명령(10212~10807 rad/s)을 받았는데도 실측은 완전히 동일. `kp_motor`를 200배 키워도 **결과가 전혀 안 변함**(여전히 7420.01) → 게인 문제가 아님.

3. **진짜 원인 확정: 모터의 물리적 토크-속도(파워 리밋) 곡선.** `Motor 1~4`의 `w_t = [0, 3750, 7500, 8000]` rad/s (`torque_speed_param = torque_power`, `power_max=160W`, `torque_max=0.8N·m` 기준 파워리밋 토크-속도 곡선의 속도 구간). 즉 이 모터는 **물리적으로 최대 8000 rad/s까지만 낼 수 있는 스펙**인데, 상위 루프가 **10,000~11,000 rad/s**를 명령하고 있어서 4개 모터 전부 "낼 수 있는 한계"에 박혀 똑같은 평형 속도로 수렴 → 차등(요) 제어 권한이 물리적으로 사라짐. 사용자가 "상한이 7450쯤으로 정의된 거 아니냐"고 짚은 게 정확히 이 현상(리터럴 Saturation 블록은 아니지만, 파워리밋 토크-속도 곡선이 사실상 같은 역할).

   **결론: 오늘 전체의 근본 원인은 결국 원래 목표였던 "바깥쪽(자세/위치) PID가 FX450 동역학에 안 맞아서 말도 안 되는 속도를 명령한다"로 귀결됨.** 프로펠러 방향/모터 서보는 부수적인 문제였고(X/Y 개선에는 실질적으로 기여), yaw 폭주의 핵심은 모터가 애초에 낼 수 없는 속도를 계속 요구받고 있었다는 것.

4. **부가 발견: 모터 스펙(`w_t` 최대 8000 rad/s)도 FX450 실물 기준으로 보면 이미 비현실적으로 높음.** FX450은 보통 **A2212 1000KV 모터 + 3S(11.1V) 배터리** 조합(무부하 최대 rpm = KV×V = 1000×11.1 ≈ 11,100rpm ≈ **1,162 rad/s**)을 씀 — 출처: [Robocraze F450 kit](https://robocraze.com/products/f450-quadcopter-frame-kit-with-a2212-kv1000-brushless-motor-and-4-30a-esc-and-2-pair-1045-propeller), [Amazon QWinOut F450 kit (A2212 1000KV)](https://www.amazon.com/QWinOut-Airframe-Quadcopter-Brushless-Propellers/dp/B08GX5Z4SN).

5. **`w_t`/`T_t`/`w_eff_vec`는 사실 죽은 파라미터였음(중요, 시간 많이 씀).** `Motor 1~4`의 `w_t`를 8000→1162.4 rad/s(FX450 실제 스펙)로 스케일링해서 재테스트했는데 **결과가 수정 전과 소수점까지 완전히 동일**(포화 속도 여전히 7420.01). `set_param` 자체는 성공(즉시 다시 읽으면 새 값 확인됨)했는데도 시뮬레이션 결과가 전혀 안 바뀜 → `torque_speed_param = ee.enum.electromech.envelope.torque_power` 모드에서는 **`w_t`/`T_t` 배열이 아예 안 쓰이고**, 대신 `qc_motor.max_power`(`power_max`)/`qc_motor.max_torque`(`trq_max`) 스칼라로 토크-속도 한계가 계산됨. `qc_motor.max_power`를 절반(160→80W)으로 줄여서 검증 → **평형 속도가 실제로 7420.01 → 5869.63으로 바뀜** (진짜 활성 파라미터 확인됨).

   **최종 결론**: 모터 속도 한계를 실제로 좌우하는 건 `qc_motor.max_power`/`max_torque`(`Electrical` 마스크의 `power_max`/`trq_max`)이고, `w_t`/`T_t`/`w_eff_vec`는 현재 설정에서 무시됨. 다만 `power_max`를 낮춰도(80W) 4개 모터가 **여전히 전부 같은 값**으로 수렴함 — 상위 PID가 여전히 1만+ rad/s를 명령하는 한, 모터 스펙만 realistic하게 낮춰봤자 "다 같이 새 한계에 박히는" 현상 자체(=차등/요 제어 상실)는 해결 안 됨. **모터 스펙 교정 + 상위 PID 재튜닝을 함께 해야 함.**

### 4차 세션: "게인을 아무리 바꿔도 yaw가 안 바뀐다" → 필터 대역폭/물리적 원인으로 재추적

`kp_attitude=5`(원래의 1/25 수준)까지 낮춰도 yaw는 여전히 369,440°(거의 그대로) — **yaw 폭주가 게인 크기와 완전히 무관하다는 게 재확인됨** (반면 roll/pitch는 초반에 한 번 크게 튄 뒤 안정화됨, 즉 실제로 게인에 반응함 — yaw만 다름).

- 프로펠러 2,3 방향 수정 + 낮은 게인 조합으로 재검증: **X/Y는 다시 거의 완벽**(거의 0 유지), **모터1&4 vs 2&3이 이번엔 실제로 살짝 다른 값**(6398.37/6379.31 vs 6398.54/6379.47 — 전처럼 4개가 완전 동일하지 않음, 차등 제어가 조금은 걸림). 하지만 **yaw는 여전히 365,507°로 초기 상태와 거의 동일**.
- **`Control Yaw`의 입력(오차 e)을 직접 로깅 → 사실상 항상 0** (~1e-8 수준), 그런데 실제 yaw는 365,507°까지 감. 게인×0=0이니 게인을 뭘 넣어도 안 바뀌는 이유가 설명됨.
- 오차 계산 체인 추적: `Add3 = Yaw Cmd(+) - Filter Yaw(-)`. `Yaw Cmd`는 `Position Control`의 **Traj(레퍼런스) 버스 Port 2**에서 온 값(우리가 고정한 `spline_yaw=0`) — 즉 목표값 자체는 정상. 문제는 `Filter Yaw`(m.Chassis.yaw 필터링) 쪽.
- **`act_yaw`(Scope 경로, 실제로는 `nYaw`=unwrap 누적값 추정) vs `filter_yaw_out`(Control Yaw가 실제 보는 값) 동시 로깅** → `act_yaw`는 369,439°까지 누적, `filter_yaw_out`은 시종일관 거의 0(-0.011°) — **두 신호가 완전히 다르게 거동**.
- **`m` 버스 자체의 배선은 정상**: `Maneuver Controller`의 Inport 1(`m`)은 `Quadcopter`(실제 물리 모델)에 직결되어 있음 — "선이 끊긴" 게 아님. `6 DOF` 블록에는 `yaw`(wrap 추정)/`yawWr`(wrap)/`nYaw`(unwrap 누적) 세 가지 버전이 존재.
- **가설(검증 필요)**: unwrap은 wrap 신호의 인위적 -180/180 점프만 제거하고 진짜 누적 회전은 보존하므로, `nYaw`가 크다는 건 **기체가 실제로 그만큼 누적 회전했다는 뜻**(허위 정보 아님). 반면 `Filter Yaw`가 받는 wrap된 `m.Chassis.yaw`는, 만약 기체가 필터 대역폭(`filtM_yaw=0.01s` 기준 ~100 rad/s)보다 훨씬 빠르게(추정 ~1484 rad/s) 실제로 회전하고 있다면, 빠르게 -π~π를 반복 순환하는 신호를 저역통과 필터가 평균 내버려 **거의 0으로 뭉개져 보이는 것**일 수 있음 — 이러면 "피드백이 끊긴 버그"가 아니라 "컨트롤러가 원천적으로 감지 불가능한 속도로 실제 물리적 회전이 일어나고 있다"는 뜻이 됨. **아직 순간 각속도를 직접 측정해서 확정하지는 못함** (6DOF 출력 버스에 각속도/omega 항목이 안 보여서 확인 방법 추가 조사 필요).
- 이 가설이 맞다면, 남은 진짜 질문은 여전히 **"왜 기체가 처음부터 그렇게 빨리 도는가"**(반작용 토크 불균형 등 물리적 원인) — 프로펠러 방향 수정으로 425k→365~380k대로 소폭 개선된 것과는 앞뒤가 맞지만, 완전한 원인 규명은 아직 못함.

### 5차 세션(오늘 마지막 확인): "롤/피치도 같은 문제 있나?" → yaw만 특이함, 빼기(오차) 로직 자체는 정상

사용자 질문: yaw만 그런 거면 이전에 짠 "오차 = 목표 - 필터출력" 계산 방식 자체가 롤/피치에도 똑같이 적용됐을 텐데, 거기서도 오차가 항상 0으로 나온다면 애초에 그 빼기 로직/탭 방식이 잘못됐다고 봐야 함. 그래서 `diagnose_yaw_reconcile.m`을 `Control Pitch`/`Control Roll`/`Control Yaw` 세 축 모두의 입력(오차)과 `Filter Pitch/Roll/Yaw` 출력을 동시에 로깅하도록 확장해서(입력 소스는 이름 추측 없이 `Line`/`SrcPortHandle` 추적으로 확정) 재검증(게인: `kp_attitude=5,ki=0,kd=2; kp_yaw=3,ki=0,kd=1; kp_altitude=0.05,ki=0,kd=0.05; kp_position=1,ki=0,kd=0.5`).

결과 (5초 호버 테스트):
- `act_yaw` (실제 물리, unwrap 누적): 마지막 6447.93 rad ≈ **369,439°** (여전히 폭주)
- `roll_error`: min=-0.086, max=**1.76098 rad(~100.9°)**, 마지막 0.460687 rad(~26.4°) — **실제로 크게 움직이고 값이 계속 변함**
- `filter_roll_out`: min=**-1.40252 rad(~-80.4°)**, 마지막 0.0598555 rad(~3.43°) — 큰 진폭에서 시작해 정착, 실제 물리 반영됨
- `pitch_error`: min=-0.00589, max=0.0134243, 마지막 0.00680731 — 절댓값은 작지만(원래 피치 게인이 낮아 응답이 작음) **0에 딱 붙어있지 않고 정상적으로 움직임**
- `filter_pitch_out`: min=-0.00965, max=0.000607, 마지막 1.64e-5 rad(~0.0009°) — 마찬가지로 작은 진폭이지만 실제 변화 반영
- `yaw_error`: min=-0.0193735, max=0.000197, 마지막 0.000197 — **시종일관 거의 0** (최대 절댓값도 ~1.1° 수준)
- `filter_yaw_out`: min=-0.000197, max=0.0193735, 마지막 -0.000197 rad(-0.011°) — **역시 거의 0**

**결론**: 롤/피치는 "오차 = 목표 - 필터출력" 탭 방식이 실제 물리 변화를 제대로 반영하고 있음(롤은 최대 100° 가까이 실제로 움직이는 게 오차에 그대로 나타남). **오직 yaw만 실제로는 369,439°까지 발산했는데 오차/필터출력 모두 최대 1° 수준에서 멈춰있음.** 즉:
- 이전에 우려했던 "빼기(오차 계산) 로직 자체가 잘못된 가정"이라는 가설은 **기각** — 탭 방식 자체는 정상.
- 문제는 **yaw 축에만 국한**된 현상이며, 4차 세션에서 세운 가설(yaw가 필터 대역폭(`filtM_yaw=0.01s`≈100 rad/s)보다 훨씬 빠르게 wrap 신호(-π~π)를 반복 순환하면, 저역통과 필터가 그 신호를 평균 내서 거의 0으로 뭉개버린다 — 롤/피치는 이런 식으로 빠르게 wrap을 반복할 만큼 빠르게 돌지 않았기 때문에 정상적으로 오차가 보임)와 정합적.
- 남은 미해결 질문(다음 세션 우선순위): (1) yaw의 순간 각속도를 직접 측정해서 이 가설을 확정, (2) yaw가 애초에 왜 그렇게 빨리 도는지의 물리적 근본 원인(반작용 토크 불균형 등, 프로펠러 `direction` 파라미터 자체는 이미 무관함이 확인됨).

**추가 메모 — "왜 3D 애니메이션(Mechanics Explorer)에서는 빙글빙글 도는 게 안 보였나?"**

사용자가 시뮬을 직접 봤을 때 관찰한 건 "시작하자마자 pitch로 확 뒤집히고 위로 계속 올라감"뿐, yaw로 계속 도는 모습은 못 봤다고 함. `nYaw`가 369,439°까지 쌓인 것과 모순돼 보이는데, 유력한 설명은 **오일러각(X-Y-Z extrinsic) 표현의 짐벌락(gimbal lock)**:

- `R to X-Y-Z Extrinsic` 블록(=`act_yaw`/`nYaw`의 출처)은 회전행렬을 roll-pitch-yaw로 분해하는데, **pitch가 ±90°에 가까워지면 roll·yaw 축이 수학적으로 얽히는 특이점**을 가짐.
- 초반의 큰 pitch 튜블(거의 뒤집힘)이 이 특이점 근처를 지나갔다면, 몸체가 실제로 빠르게 도는 게 아니어도 그 순간 yaw 각도(및 이를 unwrap 누적한 `nYaw`)가 수학적으로 폭주하듯 튈 수 있음.
- 즉 `nYaw` 369,439°는 "몸체가 실제로 그만큼 회전했다"가 아니라 "초반 튜블 중 오일러각 계산 자체가 특이점 근처에서 깨지면서 나온 수치"일 가능성이 높음 — 4차 세션에서 세운 "yaw가 실제로 빠르게 돈다"는 전제를 재검토해야 할 수도 있음.
- **검증 방법(다음 세션)**: `nYaw`가 급증하는 시점의 pitch 값이 ±90° 근처인지 확인. 맞다면 근본 원인은 "반작용 토크 불균형"보다 "초반 pitch 튜블 자체를 막는 것"(= 위쪽 자세 PID/모터 스펙 문제, 이미 알려진 원인)에 더 무게가 실림.

**오늘 세션은 여기서 종료.** (→ 아래 6차 세션에서 이 결론의 상당 부분이 뒤집힘/정정됨. 다음 세션은 6차 세션 끝의 "다음에 시도해볼 것"부터 읽을 것.)

## 6차 세션 — 결정적 재발견: "yaw 폭주"는 가짜였고, "roll 고쳤다"도 사실은 추락 상태였다

**다음 세션(특히 새 모델/새 컨텍스트)은 이 섹션부터 읽으면 됨. 위 3~5차 세션 내용은 대부분 잘못된 전제 위에서 진행된 기록이라 참고만 하고, 아래가 현재 진짜 상태임.**

### (A) "act_yaw"는 실제로 `Prop1.w`(모터1 각속도)였다

`diagnose_hover_attitude.m`이 `Scope/In Bus Element8, 9, 11`을 각각 roll/pitch/yaw로 **가정만 하고 검증 없이** 3~5차 세션 내내 재사용했다. 실제로 모델을 덤프해서 확인한 결과(`diagnose_gimbal_lock_check.m`의 Scope In Bus Element 전수 조사):

- `In Bus Element2` = `Chassis.pz` (고도)
- `In Bus Element3` = `Chassis.pitch`
- `In Bus Element4` = `Chassis.roll`
- `In Bus Element5` = `Chassis.yaw`
- `In Bus Element8` = `Prop3.thrust` (roll이 아님!)
- `In Bus Element9` = `Prop4.thrust` (pitch가 아님!)
- `In Bus Element11` = `Prop1.w` (yaw가 아님! 모터1 각속도, rad/s)

즉 "act_yaw = 369,439°"는 모터1의 각속도(rad/s, 정상적으로 수천 대)를 그대로 `rad2deg`를 씌워서 "각도"로 잘못 읽은 것. **yaw는 처음부터 폭주한 적이 없었다.** 진짜 `Chassis.yaw`(`In Bus Element5`)로 재확인(`diagnose_real_chassis_yaw.m`)한 결과 -0.011°~1.11° 사이에서 시종일관 거의 0 — `Control Yaw`의 오차(`yaw_error`/`filter_yaw_out`, Line/SrcPortHandle로 배선 추적한 신뢰 가능한 값)와 정확히 일치. 짐벌락 가설, 필터 대역폭 가설 등 4~5차 세션의 모든 yaw 관련 논의는 **애초에 설명할 대상이 없었던 상황을 설명하려던 것**이라 전부 폐기.

같은 탭 방식(`In Bus Element8,9`)을 쓴 3차 세션의 원래 roll/pitch 수치(478.96°)도 마찬가지로 의심스러움(실제로는 `Prop3.thrust`/`Prop4.thrust`였을 가능성) — 재검증 안 됨.

### (B) roll -90° 드랍은 진짜였고, "프로펠러 방향 수정으로 고쳤다"는 것도 사실은 추락 상태에서 측정한 것

`Control Roll`/`Control Pitch`의 오차를 `Line`/`SrcPortHandle`로 직접 추적한 값(`roll_error`, `pitch_error` — 이건 Scope 탭과 무관해서 신뢰 가능)과, `In Bus Element4`로 검증한 진짜 `Chassis.roll`을 다시 확인(`diagnose_real_chassis_yaw.m`, `diagnose_roll_vs_propdir.m`):

- **보수적 게인(`kp_attitude=5,ki=0,kd=2` 등) + 프로펠러 방향 수정 없음** 상태에서 실제 roll이 -89.97°까지 크게 떨어졌다가 3.43°로 정착 — 이건 진짜 물리 이벤트(`roll_error` max 100.9°와 정합).
- 프로펠러 2,3의 `direction`을 `Negative`로 바꾸면(3차 세션에서 X/Y 드리프트 해결에 썼던 그 수정) roll이 -90°→최대 2° 미만으로 거의 완전히 사라지는 것처럼 보였음.
- **그런데 이게 함정이었다.** 각 프로펠러 추력을 직접 로깅해보니(`diagnose_counter_rotation_fix.m`):
  ```
  T1=+8.359N, T2=-8.359N, T3=-8.359N, T4=+8.359N  → 합계 0.00N
  ```
  프로펠러 2,3이 모터가 물리적 최대 속도(7420 rad/s)로 돌아도 **추력을 아래 방향으로** 내고 있었다. `direction` 마스크 파라미터는 CW/CCW를 구분하는 게 아니라 **추력 부호 자체를 뒤집는 파라미터**였던 것. 결과적으로 순추력이 정확히 0이 되어 드론은 `z=0.144m`(사실상 땅에 붙은 높이)에서 뜨지도 못하고 그대로 있었을 뿐이고, roll이 안정돼 "보인" 건 제어가 잘 된 게 아니라 **애초에 아무 것도 안 움직이는 정지 상태**였기 때문. pitch가 괜찮았다는 판단도 같은 이유로 재검증 필요.

### (C) 진짜 수정: 프로펠러 방향 + 모터 회전방향(Mixer 부호)을 함께 뒤집어야 실제로 뜬다

`diagnose_counter_rotation_fix.m`에서 프로펠러 2,3의 `direction=Negative`는 유지하고, **Motor Mixer의 `Add5`(w2)/`Add7`(w3) 부호까지 전체 반전**(`+-+-` 식으로 뒤집어서 모터 자체가 반대로 돎):

```
Add5(w2): --++ -> ++--
Add7(w3): ++++ -> ----
```

결과:
- 추력: T1=7.243N, T2=7.754N, T3=7.304N, T4=7.706N — **전부 양수**, 합계 30.01N (필요 추력 약 22.3N 이상 확보)
- **z가 실제로 1.0m 근처까지 올라가서 유지됨** (t≈2.1s에 1.0m 도달, 이후 0.98~1.11m 사이에서 유지) — **세션 통틀어 처음으로 진짜 호버 비슷한 결과가 나옴.**
- 다만 **roll이 새로 ~40°(last=40.72°, maxabs=43.28°) 오프셋으로 정착** — 대각선 페어링(어느 프로펠러 쌍이 서로 반대로 돌아야 토크가 상쇄되는지)이 이 조합에서 안 맞아서 생긴 잔여 토크로 추정. 모터 속도도 W1=6907, W2=7146.3, W3=6935.8, W4=7124.2로 확연히 갈라짐(대각 쌍끼리 안 맞음을 시사).

### (D) 호버 피드포워드(`Bias Chassis`=700) 구조 파악 (진행 중, 미완결)

`Altitude and YPR Control/Subsystem` 내부 배선 확인(`diagnose_bias_subsystem.m`):
```
cmd(Control Thrust 출력) -> Bias Chassis(+700) -> Sum(++) -> Speed(gain 2*pi) -> Out1 -> Product, Thrust Control
load(m.Load.status)      -> Bias(+1) -> Bias Load(gain 260*pkgSize(1)^3*pkgDensity) -> Sum
```
즉 `Bias Chassis=700`은 **"평균 모터 회전수(rev/s로 추정) 700 + 패키지 유무에 따른 보정"**을 더한 뒤 `2*pi`를 곱해 rad/s 피드포워드로 변환하는 구조 — MathWorks 원래(더 가벼운) 기체 무게 기준 상수로, FX450(1.2726kg, `drone_mass`)에 안 맞을 가능성이 높음. `diagnose_hover_ff_fix.m`에서 실측 기반 재계산 시도했으나 **[2]단계(Bias=800 민감도 측정)에서 dw/dBias=0으로 나와** (이미 모터가 물리 한계 7420 rad/s에 포화된 상태에서 측정해서 민감도가 0으로 나온 것으로 추정) 역산값이 터무니없는 음수로 나옴 — **이 스크립트는 미완성/실패, 재작업 필요**. (C)의 올바른 카운터로테이션 수정을 먼저 적용한 뒤(포화 안 걸린 정상 동작점에서) 다시 민감도를 측정해야 함.

### (E) 확인된 블록 경로/구조 (다음 세션 시행착오 줄이기용 레퍼런스)

- 프로펠러 방향: `quadcopter_package_delivery/Quadcopter/Propeller {1..4}/Thrust and Drag/Aerodynamic Propeller` (마스크 파라미터 `direction`, `sdl.enum.PropellerDirection.{Positive,Negative}`) — **library link 안이라 set_param 시 경고 뜨지만 정상 동작(오버라이드)**.
- Scope의 실제 In Bus Element 매핑(전수 확인됨): `E2=Chassis.pz, E3=Chassis.pitch, E4=Chassis.roll, E5=Chassis.yaw, E6=Prop1.thrust, E7=Prop2.thrust, E8=Prop3.thrust, E9=Prop4.thrust, E10=Prop2.w, E11=Prop1.w, E12=Prop3.w, E13=Prop4.w` (E14~E30은 vx/vy/vz, 모터 전류, 배터리 SOC 등). **E16=yaw(레퍼런스 bare, Chassis 접두어 없음), E21=roll, E22=pitch, E23=vel, E15=pos는 Traj/레퍼런스 버스 쪽.**
- Motor Mixer(`Maneuver Controller/Motor Mixer`) 내부: `Add4`(w1,`+--+`), `Add5`(w2,`--++`), `Add6`(w4,`-+-+`), `Add7`(w3,`++++`) — Roll/Pitch/Yaw/Thrust 4개 From 신호를 조합. **w1=Add4, w2=Add5, w3=Add7, w4=Add6 (Add6/7이 서로 바뀐 이름 순서라 혼동 주의)**.
- 호버 피드포워드: `Maneuver Controller/Altitude and  YPR Control/Subsystem/Bias Chassis`(값 700) — `quadcopter_package_parameters.m`의 `drone_mass=1.2726`(FX450 실측 질량으로 이미 갱신돼 있음, 참고).
- `To Workspace`는 **모델을 저장하지 않는 한 세션 종료 시 전부 사라짐** — 매 스크립트가 자기 태핑을 처음부터 다시 만들어야 함(기존 관례 그대로 유지). `SaveFormat`은 **`Array` 대신 `StructureWithTime`을 쓸 것** — 서로 다른 To Workspace 블록끼리 시간축을 따로 맞추다가(`sim_time` 별도 Clock 태핑 등) 정렬이 어긋나는 버그가 실제로 발생했음(이번 세션 `diagnose_roll_vs_propdir.m`에서 겪음). `StructureWithTime`은 `.time`과 `.signals.values`가 한 변수 안에 항상 정확히 짝지어짐.
- 인코딩 참고: MATLAB `-batch`가 출력하는 한글이 콘솔 로캘(CP949) 때문에 파일에서 깨져 보이는 건 정상(mojibake) — 기능엔 문제없음, 그냥 영어로 로그를 찍는 게 읽기 편함.

## 7차 세션 — "roll만 불안정한가?" 검증: pitch도 똑같이 불안정함을 확인 (CAD 비대칭 가설 약화)

`Control Pitch`/`Control Roll`의 게인·필터·부호를 직접 비교(마스크 덤프)한 결과 **완전히 동일**(둘 다 부모 `Altitude and YPR Control`의 같은 `attitude_kp/ki/kd/f/limit`를 공유) — 로직/게인 비대칭은 기각. CAD 쪽(arm 배치, 관성)도 찾아봤으나 `find_system`이 라이브러리 링크 내부를 잘 못 읽어서(`FollowLinks`,`LookUnderMasks` 다 켜도 파라미터 이름 추정 실패) 확정 못함.

대신 **회전 자유도를 담당하는 `Quadcopter/6 DOF/Joints/Spherical Joint`의 `PositionTargetRotationSequenceAngles`(오일러 시퀀스 XYX, `[roll, pitch, yaw]` 순서로 확인됨)를 이용해 초기 자세에 직접 각도를 주입**해서 검증(`diagnose_pitch_ic_test.m`, 보수적 게인 `kp_attitude=5` 등 동일):

- **IC=[0,10,0]°(pitch만 +10°)**: pitch가 -0.2°~**+88.8°**까지 튀었다가 0으로 정착 — **롤과 거의 같은 규모로 발산.** 이때 roll도 평소처럼 -89.4°까지 떨어짐(pitch 외란과 무관하게 항상 발생).
- **IC=[10,0,0]°(roll만 +10°)**: roll -89.97°(평소와 동일), pitch도 -27.8°까지 흔들림.

**결론: pitch는 원래 안정한 게 아니라, 기준 호버 테스트(IC=0)가 우연히 이 불안정을 건드리지 않았을 뿐.** 아주 작은 초기 각도만 줘도 pitch도 롤 못지않게(최대 88.8°) 발산한다 — **이건 "roll만의 문제"가 아니라 자세 제어 루프 전체가 큰 각도 영역에서 근본적으로 불안정하다는 뜻**이며, CAD 좌우 비대칭 가설보다는 "게인이 FX450 동역학/큰 각도 영역에 안 맞아서 사소한 섭동도 크게 증폭된다"는 원래(1~3차 세션) 가설을 다시 강하게 지지함. 롤이 IC=0에서도 매번 -90°대로 떨어지는 건, 아마 이/착륙(초기 고도 상승) 과정 자체가 이미 작은 섭동을 만들어내고, 이게 pitch보다 roll 쪽 수치적/동역학적 경로로 먼저 증폭되기 때문으로 추정(완전히 확정은 아님).

### 다음에 시도해볼 것 (7차 세션 기준, 이게 최신)

1. **최우선**: 자세 루프(`attitude_kp/ki/kd`)가 큰 각도(수십 도)에서도 안정적으로 감쇠하도록 재튜닝. 지금(`kp_attitude=5,kd=2`)도 작은 IC 외란(10°)에 90°에 근접하게 증폭되는 걸 보면 감쇠가 전혀 안 되고 있음 — 게인을 더 올리거나(단, 모터 포화 안 걸리는 한도 내에서) 구조를 재검토해야 함.
2. (C)에서 나온 roll ~40° 오프셋 원인 확정: 대각선 페어링을 CAD/GUI로 직접 확인 후 `direction`+Mixer 부호 조합 재점검.
3. (D) 호버 피드포워드(`Bias Chassis`) 재보정: 포화 안 걸리는 상태에서 Bias↔모터속도 민감도 재측정, FX450 무게(`drone_mass=1.2726kg`+패키지) 기준으로 재계산.
4. 위 1~3이 끝나면 pitch/roll/yaw 전부 "실제로 뜬 상태"에서 IC=0 기준 재검증 (지금까지의 "정상" 판정은 대부분 추락 상태이거나 큰 IC 섭동 없이 우연히 안 튄 경우라 완전 신뢰 불가).
5. 안정적인 호버가 확인되면 그제서야 실제 PID 게인 튜닝 → `run_and_log.py` 전체 파이프라인으로 실제 궤적 테스트.
6. `Altitude and YPR Control`의 "InputName" `systune` 에러는 여전히 미해결이지만, 이제 **호버 테스트 기반 수동/반자동 튜닝**이 훨씬 실용적인 경로로 보임(우선순위 낮음).

## 8차 세션 — Kthrust/Kdrag 실측 보정, 그리고 "위치게인이 아니라 anti-windup + 축간 커플링"으로 재수렴

### (A) 프로펠러 물리상수 실측 검증
- **`Kthrust=0.1072`은 정확함** — APC 10x4.5MR 실측 정출력(static thrust) 데이터(`https://www.apcprop.com/files/PER3_10x45MR.dat`)로 RPM 4000~12000 구간 kT를 역산하면 0.1068~0.1098로 모델값과 1% 이내 일치. (중간에 "38배 차이난다"는 결론을 냈다가 — 800g 실측치가 무부하 최고속도와 잘못 짝지어진 값이었음을 발견하고 철회. 단위/공식은 MathWorks 공식 문서로 재확인, 버그 아님.)
- **`Kdrag=0.01`은 틀렸음, 실측값은 `Kdrag≈4.222841`** — `qc_motor.max_power`를 임시로 낮춰서 모델의 평형속도를 APC 데이터가 실측된 RPM 구간(1000~15000)으로 끌어들인 뒤, 같은 RPM에서 실제 토크와 모델 토크를 직접 비교(외삽 없이)해서 산출(비율 422.28배). 이 값 적용 시 pitch IC=10° 테스트의 오버슈트가 **88.8°→10.7°로 극적으로 개선** — 자세루프 불안정의 상당 부분이 이 물리상수 오류 때문이었음을 시사.
- 이에 맞는 호버 피드포워드: `n_hover = sqrt(T_need/(Kthrust·air_rho·D⁴))` (T_need=총질량·9.81/4) → `Bias Chassis ≈ 100.98` (기존 700에서 재계산).
- **미해결**: Kdrag 수정 후 기존 게인/피드포워드로는 아예 이륙을 못 함(z가 0.14m에 멈춤, 모터 405rad/s=3870RPM). Bias Chassis를 700→100.98로 바꿔도, 게인을 10배 올려도 **결과가 완전히 동일**(z, roll, Prop1.w 전부 bit-identical)한 미스터리 발견 — `Pause Motor`는 원인 아님을 확인(En=1, y=0 상수, `diagnose_pause_motor_trace.m`), `ref` vs `meas` 비교(`diagnose_ref_vs_meas.m`)로 명령 자체가 바뀌는지 보려 했으나 포트 배선 버그로 스크립트 미완성 — **원인 미확정으로 남음.**

### (B) `Position Control` 체인 구조 완전 규명
회로: `Pos Error(World)` → `PID Controller`(P=I=D=`kp/ki/kd_xy2ypr`) → `Matrix Multiply`(R_BI 회전) → `Filter` → `Dir P/Dir R`(±1/9.81, 가속도→각도 소각도 근사) → `Err2P/Err2R`(게인=`err2rp`) → `Pitch/Roll Limit`(±π/3=±60°, 스크린샷으로 확정) → `Pitch/Roll Cmd`.

- `kp_xy2ypr`/`ki_xy2ypr`/`kd_xy2ypr`/`err2rp`는 **`Position Control` 서브시스템 자신의 마스크 파라미터**이고, 각각 `kp_position`/`ki_position`/`kd_position`/`pos2attitude`(=2.4)를 그대로 가리키는 별칭(alias)임을 확인(`diagnose_kpxy2ypr_causality.m`). 즉 우리가 계속 바꿔온 `kp_position` 등은 실제로 살아있는 게인이 맞음(한때 "죽은 변수 아니냐" 의심했으나 기각).
- `err2rp=pos2attitude=2.4`는 `Dir R`의 물리적으로 정확한 `1/9.81`(가속도→소각도) 변환 뒤에 추가로 곱해지는 임의 계수 — 전체 오픈루프 게인이 `kp_position(8)×1/9.81×2.4 ≈ 112°/m`으로 상당히 공격적임을 계산으로 확인.

### (C) 그런데 게인을 아무리 바꿔도 결과가 안 바뀜 — anti-windup 누락이 진짜 원인
- `kp_position`을 0.08~80(1000배), `err2rp`를 2.4~0.2(12배), 궤적 램프를 0~2초로 바꿔가며 베이스라인 호버(IC=0)를 재실행 — **전부 결과가 사실상 동일**(roll min -89.98~-89.99°, last=4.175° 소수점까지 일치, y 5.76~5.86m 이탈). (`diagnose_pos_gain_baseline.m`, `diagnose_err2rp_sweep.m`, `diagnose_ramped_trajectory.m`)
- 원인: `PID Controller`(Position Control 내부) 마스크가 **`LimitOutput=off`, `AntiWindupMode=none`**으로 확인됨. 포화(`Roll Limit` ±60°)는 PID 블록 "바깥"에서 일어나서 PID 자신은 자기 출력이 잘리는 걸 모름 → 실제 자세가 60°를 못 넘어 오차가 안 줄면 PID 내부값이 무한정 커짐. 실측: `err2r_out`(포화 직전)이 t=1.5s에 이미 **432°(계속 증가 중, 최댓값 721°)** — 이러니 게인을 어떤 배수로 줄여도 "포화에 도달하는 시점"만 늦출 뿐 "포화된 채 눌러붙는" 결과 자체는 못 바꿈. (`diagnose_poscontrol_chain_trace.m`)

### (D) 목표위치 자체는 정상, 문제는 고도 실패가 먼저 + Y오차가 원인이 아님
- 목표위치(`Traj.pos`, Scope의 `pos` 채널)는 t=0부터 끝까지 정확히 `(0,0,1.0)`으로 완벽히 고정 — 궤적 생성 쪽엔 버그 없음. (`diagnose_target_vs_real_pos.m`)
- 드론은 t=0에 이미 z=1.0(목표)에서 시작하는데, **고도가 먼저 무너져서 t=0.45s에 z=0.14까지 급락**. y는 t=0.9s까지 정확히 0으로 깨끗하다가 그 이후 발산 시작 — 즉 "위치 오차가 커서 자세가 튄다"가 아니라 **고도 실패가 먼저, y/roll 발산은 그 다음**.
- 램프 궤적 + 오차/cmd 동시 로깅(`diagnose_ramp_error_cmd.m`) 결과: **t=0.9~1.0s 구간에서 errY는 1.6~5cm에 불과한데 cmd_roll은 이미 27°→60°(포화)로 치솟음.** Y오차가 cmd_roll 폭주의 원인이 아니라는 뜻.
- 대신 cmd_roll이 커지는 시점(t=0.3~0.4s)이 **World-frame PID의 Z채널 출력(`pidZ`)이 급증하는 시점과 정확히 일치**(pidZ 0.6→11.16, 같은 구간 pidY는 계속 0.12 이하). `Matrix Multiply`(R_BI 회전)를 통해 Z채널 출력이 Roll 채널로 새어 들어오는 축간 커플링 의심(`diagnose_zcoupling_check.m`) — 정확한 배수 관계는 아직 안 맞아떨어져서 `Matrix Multiply`/`Filter` 내부 확인이 다음 단계로 필요.

### (E) anti-windup/에러 클램프는 전부 헛다리 — 근본 원인은 결국 Kdrag였음이 최종 확정
- **anti-windup 시도**(`LimitOutput=on`, `AntiWindupMode=back-calculation`, 상한 ±50)를 걸어봤으나 **효과 없음**(roll min -89.978, 이전과 사실상 동일) — 제한값(±50)이 `1/9.81×err2rp` 배율을 거치기 전 단계라 너무 느슨해서 실질적으로 안 걸린 것으로 추정. (`diagnose_antiwindup_fix.m`)
- **Pos Error 자체에 하드 saturation**(`Subtract2`→`PID Controller` 사이에 Saturate 블록 삽입)을 ±0.5m→±0.1m→±0.01m까지 좁혀가며 테스트 — **셋 다 roll이 거의 동일하게 -89.98~-89.99°까지 튐** (`diagnose_poserror_saturate*.m`). 특히 ±0.01m(cmd_roll을 사실상 0~2° 수준으로 억제)에서도 real_roll이 t=1.3s에 -84.8°까지 튀는 게 확인되어, **cmd_roll 크기와 roll 발산이 완전히 무관함**이 가장 깨끗하게 증명됨.
  - 부가 발견: 하드 Saturation을 미분항 있는 PID 앞에 두면 클램프 경계를 넘나들 때(dt=0.01s 안에서) 신호가 불연속으로 꺾여 미분 킥을 만들어낼 수 있음(사용자 지적) — 값을 얼마로 잡든 이 문제는 못 피함. 값 제한(Saturation)이 아니라 변화율 제한(Rate Limiter)이 올바른 도구.
  - cmd_pitch가 4° 이하로 작은데도 real_pitch가 -77.6°까지 튄 사례도 확인 — **명령 크기와 실제 반응이 완전히 분리**돼 있다는 결정적 증거.
- **잠정 결론(⚠️ 아래 단서 있음)**: 오늘 시도한 anti-windup/에러클램프/축간커플링 조사는 전부 부수 현상이었고, Kdrag(0.01→4.222841 실측 보정)를 적용하면 roll/y가 정상화되는 것으로 보임:
  - Kdrag=4.222841 + `Bias Chassis` 재계산(≈100.98) + Pos Error ±0.01m(잔존 조건) 상태로 베이스라인 호버(IC=0) 재실행
  - roll: max 1.703°, last 0.793° (이전 -89.982°에서 크게 개선), y: max 0.001m (이전 5.772m에서 크게 개선)
  - **⚠️ 단서(사용자 지적, 중요): 이 테스트에서 z가 0.1444에 멈춰있음 — 즉 드론이 실제로 1.0m까지 오르내리며 움직인 게 아니라 거의 제자리에 정지한 상태였음.** Kdrag가 틀렸던 케이스들은 z가 0→1.0 사이를 실제로 오르내리는 강한 자극이 있었던 반면, 이 "성공" 케이스는 자극 자체가 약해서(드론이 안 움직이니 자세루프를 흔들 일이 없어서) roll이 작게 나온 것일 수 있음. **공정한 비교가 아니므로 "Kdrag 수정으로 roll 문제 해결"은 아직 확정이 아니라 조건부(제자리 상태 한정) 결과임.**
  - **진짜 검증은 z 이륙 문제가 먼저 풀려서 드론이 실제로 1.0m까지 뜬 상태에서 roll이 계속 깨끗한지 재확인해야 완료됨.**

### 다음 세션 지시사항 — 진단 끝, 이제 바로 실행할 것 (사용자 확정 지시, 9차 세션은 이걸로 시작)

**핵심 진단은 오늘(8차 세션)로 끝났다. 자세(roll/y) 불안정은 Kdrag 수정으로 개선되는 것으로 "보이지만", 그 테스트가 드론이 제자리에 멈춘(z=0.1444) 약자극 상태에서 나온 결과라 아직 조건부 확인일 뿐임(⚠️ (E) 단서 참고). 남은 핵심 작업은 z가 0.1444에서 못 뜨는 것 하나이고, 사용자 판단으로 이건 "동작점(트림) 설정 + 피드포워드 조정" 문제로 좁혀짐. 다음 세션은 더 진단하지 말고 아래를 바로 실행해서 끝낼 것:**

1. **[바로 실행] 동작점(트림) 제어 + 피드포워드 직접 조정**: Kdrag=4.222841 상태에서 `Bias Chassis`(현재 재계산값 100.98)를 기준으로 미세 조정하면서, 드론이 실제로 z=1.0m까지 상승하는 동작점을 찾을 것. 게인(kp_altitude 등)이 아니라 **트림/피드포워드 쪽을 우선 조정**(사용자 판단). 필요하면 `Simulink.findop`/trim 함수로 정식 operating point를 잡아도 됨.
2. 이전에 있었던 "Bias Chassis를 바꿔도 결과가 bit-identical"이던 미스터리가 재발하면, 그때 `diagnose_ref_vs_meas.m`의 포트 배선 버그를 고쳐서 `ref`(명령)가 실제로 바뀌는지부터 확인(Pause Motor는 이미 원인 아님으로 기각됨).
3. **[핵심 재검증, 빠뜨리지 말 것]** z=1.0m까지 실제로 뜨는 게 확인되면, **그 실제 비행(제자리 정지가 아닌) 상태에서** roll/y도 계속 깨끗한지 재검증할 것 — 오늘 확인된 "roll 정상화"는 드론이 거의 안 움직인 상태에서 나온 결과라 진짜 검증이 아직 안 끝났음. 이게 확인돼야 자세/고도 안정화 작업이 진짜로 마무리됨.
4. anti-windup/Pos Error saturation은 더 이상 필요 없어 보임(Kdrag 수정만으로 roll/y는 이미 해결됨) — 재도입할 필요 없음.
5. 안정화 마무리되면 실제 항속 성능(최대 수평속도) 계산: 총추력-중량 여유분과 항력계수(`qd_drag.Cd_X=0.35`, 전면적 `qd_area.XZ=0.09`)로 종단속도 추정 → `path_time.py` 속도 프로파일 상한 설계에 반영.

## 9차 세션 — 모터 포화의 진짜 정체, 추력 공식 1/91 발견, 그리고 첫 현실물리 비행

### (A) "bit-identical 미스터리" 완전 해결: 모터 PID 출력 포화(limit_motor)였다
- Control1(모터 PID)의 입력 ref/meas를 직접 태핑(`diagnose_motor_ref_trace.m`): **ref는 Bias에 따라 잘 바뀌는데(8663 vs 39444 rad/s), 모터 실측은 두 경우 다 정확히 405.32 rad/s** — 명령 전달은 정상, 모터가 명령을 못 따라가는 것.
- `diagnose_control1_output.m`: 모터 PID 출력(토크 명령)이 시작부터 끝까지 **0.2500에 완전 포화** (`max_trq_cmd = limit_motor = 0.25`). 오차가 8000 이상이라 P항만으로 한계의 100배를 넘어 항상 포화 상태.
- `limit_motor` 스윕(`diagnose_limit_motor_sweep2.m`): 0.25→405 rad/s, 0.40→512.78 rad/s (**평형속도 ∝ sqrt(limit) 정확히 성립** = 평형점에서 프로펠러 저항토크 = limit_motor 값). 0.8/1.0/2.0은 전기 솔버 붕괴(Motor 1~4 수렴 실패).

### (B) Kdrag 재재보정: 4.2228도 틀렸다, 실측 0.597
- (A)의 "평형속도 = 토크 측정기" 발견으로 직접 역산: 모델의 프로펠러 토크가 405 rad/s(3870 RPM)에서 벌써 0.25 N·m인데, 실제 APC 데이터는 같은 RPM에서 ~0.052 N·m — **Kdrag=4.2228은 5~7배 과대**(8차의 422배 보정은 거대 외삽+선형성 가정 결함).
- 재보정: 호버 속도(634 rad/s)에서 실측 토크(0.0865 N·m)와 일치하는 **Kdrag=0.597** 도출.
- **검증 성공**(`diagnose_kdrag_recalibrate2.m`): limit_motor=0.0865로 놓자 평형속도 632.17 rad/s (예측 634, 오차 0.3%). **Kdrag=0.597 확정.**
- 이 값이면 호버 필요토크 0.0865 < limit_motor 0.25, 필요파워 55W < 160W — **모터 스펙 안에서 호버가 물리적으로 가능**해짐 (limit_motor 변경 불필요).

### (C) 추력 공식 1/91 발견: 블록용 Kthrust는 9.79
- Kdrag=0.597로 모터가 1076 rad/s까지 돌아도 **추력이 프롭당 0.176N**(표준 공식 기대값 16N의 정확히 1/91). 여러 속도(405/1076/1132 rad/s)에서 비율 일정 — Aerodynamic Propeller 블록 내부 공식이 표준 `T=kT·ρ·D⁴·n²`와 1/91 다름.
- Kthrust를 38배 올렸던 옛 테스트에서 추력이 정확히 38배 커진 것 확인 → **추력 ∝ Kthrust 선형**. 따라서 블록용 올바른 값은 **Kthrust = 0.1072 × 91.3 ≈ 9.79** (실제 634 rad/s에서 5.58N/prop 재현). "38배는 잘못"이라던 8차 결론은 반만 맞았음 — APC 계수 자체(0.1072)는 표준 공식용으로 맞지만, 이 블록 안에서는 9.79를 넣어야 같은 물리가 표현됨.
- **원래 모델(Bias=700, Kdrag=0.01, 모터 6000~7000 rad/s)은 이 1/91 페널티를 비현실적 고속으로 뚫도록 설계**된 것이었음 — 지금까지의 모든 혼란의 최상위 원인.

### (D) 첫 현실물리 비행 + 추력 경로 재스케일
- Kthrust=9.79 + Kdrag=0.597 + 프롭2,3/믹서 반전(6차 수정, 저장 안 돼 있어서 7~8차 테스트에 전부 빠져 있었음!) 조합으로 **처음으로 현실적 물리에서 이륙 성공** (z가 1.0~1.19에서 ~2초 비행 후 자세 불안정으로 추락). (`diagnose_final_calibration.m`)
- 모터 ref 인플레이션(~8663 rad/s)의 구성 분해(`diagnose_ref_composition.m`): **ref = 2π × (고도PID출력 + Bias Chassis + (load+1)×260×패키지질량)**. `Speed` 게인=2π(단위변환, 정상), **`Bias Load`의 260이라는 상수도 옛 추력맵 기준**. 고도 PID 출력이 992까지 폭주(보수 게인으로 설명 불가능한 크기, 원인 미상).
- 재스케일(`diagnose_thrust_path_rescale.m`): Bias Chassis 700→**56.5**(기체분), Bias Load 260→**44.4**(패키지분), 고도PID 출력에 ±30 rev/s 클램프 → **모터 포화 해소 성공**(W가 460~852에서 실제로 변동). 단 자세는 여전히 전복.

### (E) 남은 문제의 정리 (다음 세션 최우선)
- **yaw "-655°"는 아마 가짜**: 자세 센서 오일러 시퀀스가 **XYX**라 roll≈90°에서 짐벌 특이점 — 애니메이션에선 실제 스핀이 안 보임(사용자 확인). yaw 게인 부호 A/B도 결과에 영향 전무(= yaw 신호가 실제 회전이 아니라는 방증).
- **Motor Mixer 채널 테이블 확정** (`diagnose_roll_sign_fix.m` 과정에서, 채널 순서 R/P/Y/T): w1=Add4(+,−,−,+), w2=Add5(−,−,+,+), w3=Add7(+,+,+,+), w4=Add6(−,+,−,+). (1,4)/(2,3)이 대각선 페어 맞음. roll 채널만 4모터 전부 부호 반전해봤으나 **결과 거의 동일** — roll 채널의 권한 자체가 전복 토크 대비 미미.
- **[확정] 모터 2,3의 반대회전이 물리적으로 실현되지 않고 있음.** `diagnose_motor_spin_signs.m` 실측: **ref2=-841 vs W2=+818, ref3=-818 vs W3=+797** — 음수 명령에 맞서 양의 방향으로 전속 회전(t=0.1s부터 이미 +812). 프롭 2,3(Negative 날개)이 정방향으로 돌면서 반작용 토크가 상쇄되지 않고 쌓임 → 스핀업 즉시(t=0.25s roll -20°) 전복의 뿌리. Negative 날개가 정방향 회전인데도 추력은 +로 나오는 것(T2=9.3N)은 블록 부호 규약상의 잘못된 자기가속 평형으로 추정. **왜 모터가 음수 ref를 추종 못 하는지가 다음 세션 1번 과제** — 후보: (a) Motor & Drive 블록의 torque-speed envelope가 양의 사분면만 정의됨(w_t=[0..8000]rpm, 단방향 드라이브 가능성), (b) 모터 PID/ref 경로 어딘가의 절대값/정류 처리, (c) 애초에 음수 속도 대신 기계적 조인트 축 미러링(Transform Propeller/Revolute Joint의 축 반전)으로 반대회전을 구현해야 하는 구조.
- attitude 권한도 새 레짐 재조정 필요: attitude_limit=800(옛 스케일)은 호버 101 rev/s 대비 과대 — 이번 세션 임시로 50 사용.

### (F) 세션 막판 대발견: 공장 반대회전 메커니즘의 전모 + 6차 수정이 추력을 뒤집고 있었다
- **믹서를 원래 부호로 되돌리자 모터 2,3이 스스로 음의 방향으로 돌았음** (ref +822 → W2,W3 = **-801**, `diagnose_mixer_revert.m`): 모델 하류(Electrical 어딘가)에 모터 2,3용 부호 반전이 **원래부터 내장**되어 있음. 6차의 믹서 반전은 이 내장 반전과 이중 반전이 되어 반대회전을 깨고 있었던 것.
- **`Thrust Direction` 게인 블록 발견** (`Propeller N/Thrust Direction`): P1=+1, **P2=-1** (P3도 -1로 추정) — 음으로 도는 프롭 2,3의 원시 추력(-)을 다시 뒤집어 위로 향하게 하는 공장 설계 장치.
- **디스크 상태의 `direction` 파라미터는 4개 프롭 전부 Positive** — 6차부터 모든 스크립트가 넣어온 "프롭 2,3 direction=Negative" 오버라이드가 공장 추력 사슬에 반전을 하나 더 얹어 **추력을 아래로 뒤집고 있었음**. 이것이 "프롭 2,3 순추력 0/음수" 미스터리의 진짜 원인.
- **오버라이드 제거(완전 공장구성) + 보정값 적용 결과** (`diagnose_factory_config.m`): **추력 4개 전부 양수 (T=8.8~9.0N, 합 35.6N > 22.3N)** — 추력/반대회전/토크상쇄 전부 정상화. 공장 사슬: `양수 믹서 명령 → 모터 2,3 내장 반전(음의 회전) → Positive 프롭 원시 추력(-) → Thrust Direction(-1) → 최종 추력(+)`.
- **그러나 roll은 공장 구성에서도 여전히 cmd와 정반대로 발산** (cmdR +22.5 → realR -24.3 @0.25s → -62.6 @0.5s → 전복). 이건 우리가 넣은 어떤 오버라이드 때문도 아닌 **모델(FX450 CAD 교체본) 원래의 결함** — 프로젝트 내내 roll이 항상 -90°로만 떨어진 근본 원인.
- **기하 180° 가설 기각**: 믹서의 Roll/Pitch 채널 부호를 4모터 전부 반전(`diagnose_geometry_180.m`)해도 **roll 발산이 완전히 동일** — roll 전복은 roll/pitch 믹서 채널을 통해 일어나는 게 아님 (채널 부호와 무관).
- **최신 가설 (미검증, 다음 세션 1번)**: 자세 센서의 오일러 시퀀스가 **XYX**라는 것이 핵심 결함일 가능성. XYX의 세 번째 각("yaw"로 사용 중)은 z축 회전이 아니라 **또 하나의 X축(roll) 회전**이며, 작은 각도에서 yaw는 XYX로 표현 불가. 즉 "yaw" 컨트롤러가 사실상 roll 신호를 보고 z축 토크를 가하면서 roll 동역학을 오염시키는 구조 → roll 채널 부호를 아무리 바꿔도 발산이 안 변했던 것과 정합. **검증 스크립트 `diagnose_yaw_off.m` 작성 완료(kp_yaw=kd_yaw=0으로 yaw 루프만 끄고 공장구성+보정값 호버) — 아직 실행 안 함, 다음 세션 첫 실행 대상.**

### (G) 통제실험 완결: "roll 발산"은 제어 루프와 100% 무관 — XYX가 yaw 스핀을 roll로 오인 보고하는 것이 최종 가설
9차 세션 막판에 공장구성(+보정값) 기준으로 아래 개입을 전부 시도했고, **모든 경우 roll 발산이 소수점 수준까지 동일**(-22.7° @0.25s → -57.7° @0.5s → 전복)했음:
- yaw 제어 완전 OFF (`diagnose_yaw_off.m`) → 불변
- 자세 게인 10배 (kp=50, kd=20, `diagnose_attitude_gain_up.m`) → 오히려 더 빨리 전복(-37° @0.25s). 권한 부족 가설 기각, "능동적 전복"임을 확인
- 믹서 roll 채널 부호 반전 / roll+pitch 동시 반전 (`diagnose_roll_sign_fix.m`, `diagnose_geometry_180.m`) → 불변
- cmd roll/pitch에 -1 (외곽 경로, `diagnose_cmd_negate.m`) → 불변 (경로가 실경로임은 배선 추적으로 확인: PC Out1/2 → YPR In3/4)
- Roll Cmd → **PID 오차합(Add2) 입력에 직접 -1 삽입** (`diagnose_pid_input_negate.m`) → 불변

**결론: 이 "roll 발산"은 제어가 만드는 운동이 아님.** 최종 가설(강력): 순수 yaw 회전(Rz)은 XYX 오일러 분해에서 **X축 각도(=Chassis.roll 신호)로 나타남** (XYX에는 Z축 성분이 없음). 애니메이션에서 관측된 "뒤집혀서 계속 돌아감"(지속 회전)과 합치면 — **기체는 상쇄되지 않은 반작용 토크로 yaw 스핀 중이고, XYX 센서가 그것을 roll로 오인 보고**하는 것. 그래서 roll 계통의 어떤 개입도 숫자를 못 바꿨던 것. 반작용 토크 불균형의 유력 원인: `Thrust Direction` 게인(-1)이 프롭 2,3의 **힘은 뒤집지만 반작용 토크는 안 뒤집는** 배관 (External Force 블록은 ForceY와 TorqueY를 둘 다 몸체에 가하는데, 토크 경로에 -1이 있는지 미확인).

### (H) 9차 세션 최종 상태: 발산 원인의 모든 1차 가설 기각 — 남은 후보 2개
(G)의 통제실험에 이어 세션 막판에 추가로 기각된 것:
- **XYX가 yaw 스핀을 roll로 오인보고한다는 가설 기각** — 사용자가 애니메이션으로 직접 확인: yaw 스핀이 아니라 **정말로 상하가 뒤집혀서(도립 자세로) 날아다님**. 즉 실제 roll/pitch 반전이 일어나는 것이 맞음.
- **cmd 크기 가설 기각** (`diagnose_cmd_cap.m`): Pitch/Roll Limit을 ±10°로, x,y sat을 ±0.05m로 조여 cmd_roll이 정확히 10°에 캡됐는데도 실제 roll은 -61°까지 동일하게 발산.
- **역진자(추력면이 CG 아래) 가설 기각** (`diagnose_prop_height.m`): 프롭 장착점을 [0,0,-20]→[0,0,80]mm(+100mm)로 올려도 발산 불변. (단, Transform 파라미터의 library-link 오버라이드가 컴파일에 실제 반영됐는지는 미확인 — direction 파라미터는 반영됨이 확인됐으므로 아마 반영됐을 것)

**종합**: cmd 부호/크기, PID 입력 부호, 믹서 채널 부호(roll/pitch/yaw), 게인 크기(10배), yaw 루프 유무, 프롭 장착 높이 — **모두 발산에 영향 없음**(-22~-25° @0.25s, -57~-63° @0.5s로 사실상 동일). "뒤집힌 자세가 안정 평형"이라는 사용자 관찰이 유일하게 일관된 사실.

**남은 후보 (다음 세션에 검증할 것)**:
1. **`Thrust Direction` 게인(-1)이 힘뿐 아니라 반작용 토크(Q)까지 곱하고 있을 가능성**: 그러면 프롭 2,3의 반작용 토크 부호가 뒤집혀 4개가 상쇄되지 않고 합산됨 → 큰 순 토크. External Force 블록은 ForceY와 TorqueY를 모두 몸체에 가하므로, `Thrust and Drag` 내부의 Drag Sensor→External Force 배관에서 Thrust Direction 게인이 정확히 어느 신호(힘만인지, 힘+토크인지)를 곱하는지 확인해야 함.
2. **자세 실측(Chassis.roll/pitch) 부호가 물리와 반대일 가능성**: cmd와 real이 늘 반대 방향인 것이 "제어가 반대로 미는" 게 아니라 "측정 부호가 뒤집혀 있어 실제로는 명령을 따르는 중인데 기록이 반대로 보이는" 경우 — 단 이러면 뒤집힌 자세가 안정 평형인 것과 정합적인지 별도 검증 필요. 확정 방법: `6 DOF/Transform Sensor`의 `SenseOmegaX/Y/Z`를 켜서 **실제 각속도 벡터**를 찍고, 애니메이션과 실측 부호를 대조.

### (I) 추가 기각: 과대 루프게인 가설도 아님 — 발산은 ~0.2 N·m급 고정 프로파일 외란 토크
- `diagnose_attitude_gain_down.m` (kp=0.5, kd=0.5): 발산 불변 (-23.2° @0.25s). **게인 0.5~50 (100배 범위)에서 초기 발산 궤적(-2.1° @0.1s → -23° @0.25s)이 완전히 동일** — 자세 PID를 통과하지 않는 토크.
- 크기 역산: 0→-2°/0.1s에서 각가속도 ~7 rad/s², 관성 ~0.03 kg·m² 기준 **τ ≈ 0.2 N·m의 지속 토크**가 t=0부터 작용. 모터 속도 편차(±1% → ~0.02 N·m)로는 불가능한 크기 → 추력 비대칭이 아니라 **직접 가해지는 토크** = External Force 블록의 반작용 토크(TorqueY, 프롭당 0.1~0.25 N·m급) 배관의 부호/좌표 문제가 유일하게 남은 후보.
- 사용자 관찰 종합: 출력이 cmd를 따라가긴 하나(추종 자체는 이루어짐) 이 외란 토크에 밀려 극단으로 튐. 뒤집힌 자세에서 안정되는 것은 뒤집히면 이 토크의 작용 방향이 반전/소멸되기 때문으로 추정.

### (J) 세션 마감 분석: cmd_roll 폭증의 진짜 경로와 클램프 설계 방침 (사용자와 합의)
- **"x,y 오차를 클램프했는데 cmd가 60°까지 커지는 게 물리적으로 가능한가?"** → 클램프된 x,y 오차만으로는 불가능함을 수치로 확인: P항 상한 = 8×0.15×2.4/9.81 = **16.8°** (0.05m 클램프면 5.6°), 클램프에 눌러붙은 동안 D항=0, I항(ki=0.04)은 0.1° 미만. **실제 경로는 8차 (D)에서 측정했던 Z채널 누출**: z 오차(추락 시 ~1m)는 클램프 밖이고, 그 큰 z채널 PID 출력이 기체가 기울면 `Matrix Multiply`(R_BI)에서 sin(기울기)에 비례해 roll/pitch 명령으로 섞임 (pidZ≈3~11, 60° 기울면 37°+ 기여 — cmd가 60을 치는 t=0.75s가 정확히 "큰 기울기+z 추락"이 겹치는 시점).
- **I항 가설 판정**: 비행 중 cmd 급증의 원인은 아님(수치상 2자릿수 부족). 단 추락 후 cmd가 0.08~0.1°/s로 천천히 흐르는 드리프트는 I항 와인딩이 맞음(계산치와 일치).
- **roll 기준입력 -1 삽입이 무효였던 이유**: 기계적으로는 삽입 성공(배선 추적 확인). 반전 전후 roll 오차가 +44.5° vs +1.5°로 30배 달라졌는데도 모터 ref가 588 vs 585로 거의 동일 — **roll PID 출력의 모터 명령 기여 자체가 외란 토크 대비 미미**해서 부호가 어느 쪽이든 물리에 흔적을 못 남김.
- **클램프 설계 방침 (합의)**: z는 고도 제어 본업(이륙/회복) 때문에 **약하게만 클램프**(예: ±0.5m)하고, Z→roll/pitch 누출은 **회전(Matrix Multiply) 이후 단의 `Pitch/Roll Limit`에서 타이트하게**(±10~20°) 자르는 구조로. (±10° 캡은 이번 세션 `diagnose_cmd_cap.m`에서 cmd를 확실히 묶는 것 확인 — 외란 토크 문제와는 별개)

### (K) 최종 가설 정련 (사용자 통찰): "고정 외란 토크"가 아니라 "도립진자형 자세 의존 모멘트"
- 상수 토크라면 기체를 계속 돌려야 하는데, 실측은 **뒤집힌 자세(~81°)에 도달하면 정확히 거기서 정지·유지** — 즉 자세에 따라 방향이 바뀌는 **복원 모멘트** (정자세에서 밀어내고 도립 자세로 끌어당김) = **진자 구조**. 무게중심이 추력 작용점보다 위 → 정자세는 도립진자(불안정), 뒤집힘이 안정 평형.
- 이 가설은 오늘 모든 관측과 정합: 제어 부호/게인/캡 완전 무관(중력 모멘트) ✓, 미세 씨앗에서 지수 성장 ✓, 발산 프로파일 항상 동일(진자 시정수 고정) ✓, 도립에서 완전 안정 ✓.
- 크기 역산: 관측 각가속도(~7 rad/s² @ 2°)에 필요한 **CG 오프셋 ≈ 26cm** — 미세 비대칭이 아니라 **1kg 패키지/배터리급 질량이 통째로 잘못된 쪽(위)에 장착된 스케일**. FX450 CAD 교체 시 패키지/프레임 장착 방향이 뒤집혔을 가능성 유력.
- (H)의 "프롭 +100mm 무효" 기각은 **유효한 기각이 아니었을 수 있음**: Transform Propeller는 프롭 몸체만 옮기고 힘 작용 프레임/CG는 안 움직였을 가능성.

### (L) 10차 세션: 사용자 결정타 — "드론 CAD 전체가 roll축으로 90° 돌아간 채 저장되어 있다"
- **사용자 확인**: FX450 CAD가 처음부터 **드론 전체가 roll 방향(YZ평면)으로 90° 틀어진 채 저장**되어 있음. (K)의 "패키지가 위에 장착됐나"보다 넓은 진술 — 기체 전부가 돌아가 있는 것.
- 이 한 가지로 지금까지의 관측 전부가 설명됨:
  - 솔리드 관성/CG 방향이 90° 돌아감 → 원래 "아래(-Z)"로 가야 할 CG 성분이 **옆(Y)** 으로 감 → 중력 × 횡방향 CG 오프셋 = **자세 의존 roll 모멘트** (실측 ~0.2~0.25 N·m ↔ 총질량 2.27kg 기준 오프셋 ~1cm 스케일)
  - roll이 -90°에 도달하면 그 CG가 진자 바닥으로 내려감 → **뒤집힌 자세가 안정 평형** = (K)의 진자 관측과 정확히 정합
  - 제어 부호/게인/캡 개입이 전부 무효였던 것 ✓ (중력 모멘트는 제어 루프 밖), 발산 프로파일 불변 ✓ (진자 시정수 고정)
- **모델 구조 증거 — 10차 후반 실측으로 대폭 교정됨**:
  - **[철회] "팔이 수직으로 서 있다"는 이전 판독은 오류.** Transform Arm1~4의 활성 회전은 `RotationMethod=RotationSequence`, **ZXZ [θ 90 0]** (θ=0/-90/+90/180) — 즉 **X90 보정이 이미 내장**되어 있음. 이전에 읽은 "+Z 회전 0/-90/90/180"은 비활성 StandardAxis 파라미터의 잔재값이었음 (교훈: RotationMethod부터 확인할 것). 다리는 StandardAxis +X 90(활성). **이 보정들은 패치가 아니라 MathWorks 원래 설계** — 원본 모델은 원래부터 '옆으로 저장된 부품'을 Transform으로 눕혀 쓰는 구조.
  - **[핵심 실측] FX450로 교체된 STEP 3종이 원본과 다른 방향으로 저장됨** (서브모듈 커밋 `156cd3b`가 교체 커밋, 직전 `bc6bb85`가 MathWorks 원본; CARTESIAN_POINT 분포 p1~p99 비교):
    - plate_bottom: 원본 얇은축 **Z**(1.5mm, XY평면에 눕힘) → FX450 얇은축 **Y**(1.6mm, XZ평면에 세워짐) = **X축 90° 회전(Y↔Z 스왑)**
