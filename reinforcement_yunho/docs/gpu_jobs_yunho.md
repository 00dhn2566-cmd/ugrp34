# GPU 클러스터 작업 지시서 — 윤호 (VRAM 40GB × 20)

> 목적: 팀 전체에서 연산량이 큰 작업을 모아 윤호의 GPU 환경에서 실행한다.
> 원칙: **빌린 GPU 시간은 학습·스윕에만 태운다.** 데이터 생성·스크립트 검증·전처리는 사전에 로컬에서 끝내 놓는다.
> 기준 문서: README.md §7, window_detection_spec_v0.2.md, overall_gilnam/vision/model_decisions.md

---

## Job 0 — 환경 확인 (착수 당일, 30분)

- [ ] `nvidia-smi`로 **GPU 기종 확인** — 40GB면 A100 계열일 가능성이 높음
  - **A100은 RT 코어가 없어 Isaac Sim의 RTX 렌더러(카메라 센서·Replicator)가 공식 지원되지 않는 것으로 알려져 있음.** 물리 전용(headless, 카메라 없는 RL)과 일반 딥러닝 학습은 문제없음
  - → A100이면: **데이터셋 생성(Replicator)은 로컬 RTX 머신에서**, 클러스터는 학습 전용으로 분리 (아래 Job 4는 조건부)
  - → RTX 계열(A40/L40S 등)이면: Job 4까지 전부 클러스터에서 가능
- [ ] CUDA/드라이버 버전, 컨테이너(Docker) 사용 가능 여부, 스토리지 용량·데이터 반입/반출 경로 확인
- [ ] 멀티 GPU 스케줄링 방식 확인 (잡 스케줄러 유무 / `CUDA_VISIBLE_DEVICES` 수동 배분)

## Job 1 — YOLO-pose 본 학습 (길남 파트 위탁) — P0, 데이터셋 도착 즉시

작고 급한 작업. 팀의 "본 학습용 CUDA GPU 확보" 안건이 이걸로 해소됨. 길남이 토이 데이터로 학습 루프 전체(데이터→학습→평가)를 이미 검증해 놨으므로 데이터셋만 있으면 바로 실행 가능.

- **입력물**: ① 1차 데이터셋(윤호 생성, ~3,000장 — `To_do_checklist_yunho.md` 2번) ② `overall_gilnam/vision/` 폴더 (window_pose.yaml, eval_corners.py, requirements.txt)
- **사전 설정**: `pip install -r requirements.txt` (**ultralytics==8.4.87 핀 준수**), `window_pose.yaml`의 `path:`에 데이터셋 절대경로 기입
- **기준 명령** (model_decisions.md):
  ```
  yolo pose train model=yolo11s-pose.pt data=window_pose.yaml imgsz=640 single_cls=True epochs=100
  python eval_corners.py --model runs/pose/train/weights/best.pt --dataset <데이터셋 경로>
  ```
- **반환물**: `runs/pose/train/weights/best.pt` + `results.csv` + eval_corners 출력 표(거리 구간별 mean/p95 픽셀 오차) → 길남
- **규모**: GPU 1장, 수 시간 이내. 40GB VRAM이면 batch를 크게 잡아도 됨
- **병렬 변형 실험** (GPU 여유분 3~5장으로 동시 실행 — model_decisions의 후속 항목들을 한 번에 해소):
  - [ ] `imgsz=960` 비교 학습 (원거리 corner 오차 과대 시 대비, #6)
  - [ ] `single_cls` 제거 3클래스 실험 (#4 재평가 — 플래그 전환만으로 가능)
  - [ ] yolo11n vs yolo11s 비교 (속도/정확도 트레이드오프 확인)
- **2차 데이터셋(정책 C) 도착 시 동일 절차로 재학습**

## Job 2 — RL 훈련 파이프라인 검증 (스모크) — P0, RL 환경 완성 직후

스윕에 20장을 태우기 전에 반드시 1장으로 먼저 검증. README §7.5의 "훈련 파이프라인 먼저 안정화" 원칙.

- **내용**: 최소 기능 보상 + GT 관측으로 end-to-end 학습이 도는지 확인 (환경 병렬화 · 로깅 · 체크포인트 저장/재개 동작 포함)
- **규모**: GPU 1장. 40GB VRAM은 병렬 환경 수(num_envs)를 크게 잡는 데 사용 (정책 네트워크 자체는 작음)
- **통과 기준**: 학습 곡선이 기록되고, 중단→재개가 되고, 체크포인트 정책이 평가 스크립트에 물려 돌아갈 것

## Job 3 — RL 보상 가중치·하이퍼파라미터 스윕 — P1, 20장의 주 소비처

README §7.5가 지목한 최대 일정 리스크 = "보상 튜닝 반복이 길어지면 방학 checkpoint 위협". 순차 반복 대신 **병렬 스윕으로 반복 시간을 압축**하는 것이 20장을 빌리는 핵심 이유.

- **스윕 축** (README §7 기준):
  - 보상 가중치: 통과 보상 / 충돌 페널티 / progress / 자세·에너지 (§7.2 — config로 분리해 두었으므로 config만 갈아끼우며 실행)
  - 커리큘럼 단계 (쉬운 배치 → 어려운 배치) 유/무
  - 관측 노이즈 수준 (§7.6 — GT → 노이즈 주입 단계별)
  - 도메인 랜덤화 강도 (§7.1)
- **seed는 config당 최소 3개** — RL은 통계적 성공률로 말해야 하므로(§7.4) 단일 seed 결과로 config 우열을 판단하지 말 것
- **배분 예시**: 6 config × 3 seed = 18장 동시 / 2장은 예비·평가용
- **반환물(라운드마다)**: config·seed·커밋 해시 / 체크포인트 / 학습 곡선 / reward hacking 여부 메모(창문 근처 맴돌기·통과 직전 회피 관찰)
- **주의**: reward hacking 판별은 곡선만으론 안 됨 — 라운드마다 대표 에피소드 몇 개는 렌더/궤적 시각화로 확인

## Job 4 — 평가 배치 (§7.4 프로토콜) — P1, 각 스윕 라운드 후

- **내용**: 스윕에서 나온 상위 정책들을 랜덤 씬 N개(수백~수천, 창문 개수·배치·색 순서 랜덤)에서 병렬 평가
- **지표**: 통과 성공률 · 충돌률 · 평균 통과 시간 / **baseline(단순 웨이포인트 추종)을 동일 씬 세트에서 함께 평가** (보고서의 "왜 이 성공률이면 충분한가" 대비)
- **규모**: GPU 2~4장 (씬 seed 분할로 병렬화)
- **반환물**: 정책별 지표 표 CSV → 팀 공유

## Job 5 — (조건부) 데이터셋 대량 렌더링 — Job 0 결과에 따라

- Job 0에서 GPU가 RTX 계열로 확인된 경우에만: 2차 데이터셋(정책 C·확장분)과 도메인 랜덤화 폭 확대분을 씬 seed 분할로 GPU당 독립 렌더 → 병합
- A100으로 확인되면 이 작업은 로컬 RTX 머신 담당으로 확정하고, 클러스터에는 올리지 않는다

## 20장 배분 계획 (예시)

| 시점 | 작업 | GPU |
|---|---|---|
| 데이터셋 완성 직후 | Job 1 본 학습 + 변형 실험 3종 | 4~6장, 반나절~1일 |
| RL 환경 완성 직후 | Job 2 스모크 | 1장 |
| 스모크 통과 후 | Job 3 스윕 (config×seed) | 12~18장, 라운드 반복 |
| 각 라운드 후 | Job 4 평가 배치 | 2~4장 |

## 운영 수칙

1. **재현성**: 모든 실행에 seed · config 파일 · git 커밋 해시를 기록. 산출물 폴더명에 포함 (예: `run_<config>_<seed>_<hash>/`)
2. **회수 목록 고정**: 잡마다 위 "반환물" 항목만 압축해 반출 — 학습 산출물 전체를 끌고 다니지 않는다 (길남 쪽 관례: 학습 산출물은 gitignore)
3. **우선순위 충돌 시**: Job 1(작고, 길남이 즉시 언블록됨) > Job 2 > Job 3. 스윕이 20장을 다 물고 있어도 Job 1은 1장 빼서 먼저 처리
4. **사전 준비 체크**: 클러스터에 올리기 전에 로컬/토이에서 스크립트가 끝까지 도는지 확인 (길남의 토이 리허설 방식과 동일) — GPU 시간에 디버깅하지 않는다
