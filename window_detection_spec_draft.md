# 창문 탐지 데이터셋 · 인터페이스 규격 (초안 v0.1)

> 목적: 이미지 처리기(창문 탐지)의 **데이터셋 생성 규격**과 **VIO 전달 규격**을 하나의 문서로 통일한다.
> 이 문서 한 벌이 (1) 윤호의 Isaac Sim 데이터 생성, (2) 류길남의 검출 모델 학습, (3) VIO 파트로의 데이터 전달을 모두 묶는 기준이 된다.
> 표기: **[확정]** = 합의됨 / **[초안값]** = 제안된 기본값, 조정 가능 / **[미정]** = 팀 결정 필요

---

## 0. 문제 세팅 요약

- 창문(사각 개구부)들이 공간에 **임의 위치**로 흩뿌려져 있다.
- 통과 **순서는 색깔로 라벨링**되어 미리 정해진다. (예: 빨강 → 초록 → 파랑 …)
- 이번 프로젝트의 실질 목표는 **시뮬레이션(Isaac Sim)** 구동이며, 하드웨어 구동은 여유가 될 때 진행.
- 데이터셋은 외부에서 찾지 않고 **Isaac Sim 내에서 자동 생성**(윤호 담당)한다.

---

## 1. 이미지 처리기의 목표 [확정]

카메라 프레임에서 **"다음에 통과할 창문이 지금 화면 어디에 있는가"**를 실시간으로 산출한다.

책임 범위:
1. 프레임 내 창문(사각 개구부) 검출
2. 창문 색 판정 → 통과 순서 식별
3. 창문의 이미지상 기하 정보(네 모서리·중심) 추출

책임 **밖** (VIO·융합단의 몫):
- 창문의 3D 공간 위치 복원
- 드론 자기 위치·자세 추정

## 2. 데이터셋 생성 규격 (Isaac Sim / 윤호 담당)

### 2.1 이미지 사양

| 항목 | 값 | 비고 |
|---|---|---|
| 해상도 | **[미정]** 640x480 / 1280x720 / 1920x1080 중 택 | 720p가 무난한 절충안. 확정 시 §4 intrinsics와 반드시 일치 |
| 카메라 intrinsics | fx, fy, cx, cy, 왜곡계수 | 데이터 생성에 쓴 카메라 값을 그대로 문서에 기록. VIO 전달 규격의 intrinsics와 **동일해야 함** |
| 컬러 포맷 | RGB | |

### 2.2 창문 배치 랜덤화 (도메인 랜덤화) [초안값]

| 항목 | 범위(제안) | 비고 |
|---|---|---|
| 창문 개수(씬당) | 1 ~ N개 | 프레임에 여러 개 동시 등장 허용 |
| 위치 | 카메라 전방 작업 공간 내 랜덤 | 실제 통과 시나리오 반영 |
| 거리 | 근/중/원거리 골고루 | 원거리 소형 창문도 학습되게 |
| 각도(yaw/pitch) | 정면 ~ 비스듬 | 기울어진 창문의 corner 학습 목적 |
| 조명 | 밝기·방향 랜덤 | 색 판정 강건성 확보 |
| 색 종류 | **[미정]** §2.4 참조 | |

### 2.3 프레임당 라벨 (핵심)

각 창문마다:
- **네 모서리 픽셀 좌표** `[(u,v) × 4]` — 정해진 순서(예: 좌상→우상→우하→좌하)로 일관되게
- **중심점** `(u,v)` — 편의 필드
- **색 라벨 / 순서 인덱스** — §2.4
- (선택) **visibility 플래그** — §2.5 결정에 따름

> corner 좌표는 창문 4개 모서리의 3D 좌표를 카메라로 투영해 자동 산출.
> ※ Isaac Sim Replicator는 bbox·segmentation은 기본 제공하나, **4-corner keypoint는 기본 출력이 아닐 수 있음** → 3D 점 투영 커스텀 처리 필요. 윤호가 최신 Replicator 문서로 확인할 것.

### 2.4 색(=순서) 정의 [미정]

- 색 개수: 3개(빨/초/파) / 4~5개 / 6개 이상 중 **택**
- 색 ↔ 순서 인덱스 매핑 테이블을 문서에 고정 (예: red=0, green=1, blue=2 …)
- 조명 변화에 강건하도록 채도 높은 원색 권장

### 2.5 부분 가림/화면 잘림 창문 처리 [미정]

세 옵션 중 택 (초기엔 A로 시작해 B/C로 확장 권장):
- **A. 제외** — 완전히 보이는 창문만 라벨. 가장 단순, 초기 검증에 적합.
- **B. 추정 좌표** — 가려진 corner도 화면 밖 좌표로 라벨. 통과 정렬엔 유리하나 라벨 노이즈 위험.
- **C. visibility 플래그** — corner별 가시성(0/1)을 함께 부여. 가장 유연, 라벨·학습 복잡도 증가.

### 2.6 규모·포맷 [초안값]

| 항목 | 값 |
|---|---|
| 데이터 규모 | 초기 수천 장 → 확장 |
| 분할 | train / val / test |
| 저장 포맷 | 이미지(png) + 라벨(json 또는 YOLO-pose txt). 학습 프레임워크에 맞춰 확정 |

## 3. 검출 모델 방침 (류길남 담당) [확정 방향]

- 방식: **4-corner keypoint 검출** (창문 네 모서리를 직접 예측)
  - 통과 자세 정렬에 필요한 corner를 바로 제공. 드론 게이트 통과 연구 계열과 정합.
  - 구현은 YOLO-pose(keypoint) 헤드를 4점으로 설정하는 방식이 현실적.
- 색 판정: 검출된 창문 영역을 잘라 **HSV 규칙 기반 후처리**로 판정 (모델 클래스로 학습하지 않음). 조명·색 변화 심해지면 클래스 학습으로 전환 검토.

### 3.1 단계적 구축 전략 [확정]

1. **파이프라인 우선 검증**: 데이터셋·모델 학습 완료 전, **Isaac Sim ground-truth corner를 그대로 흘려보내** 전체 흐름(탐지 → VIO 융합 → 경로계획 → 제어)이 끝까지 도는지 확인.
2. **데이터셋 생성**(윤호) 병행.
3. 데이터 확보 후 **4-corner 모델 학습** → ground-truth 자리를 학습 모델 출력으로 교체.
4. corner **정밀화**는 별도 트랙으로 진행.

> 규격을 처음부터 corner 기준으로 고정 → 나중에 box→corner 갈아엎는 재작업 없음.

## 4. VIO 전달 규격 (프레임 단위 출력) [초안]

이미지 처리기 출력은 **"관측 순간의 드론 pose(VIO)"와 시간 정렬**되어야 3D로 복원 가능. 따라서 동기화 정보가 필수.

### 4.1 프레임 단위 메시지 구조

```
{
  "timestamp": <int, ns>,        // 카메라 프레임 취득 시각 (VIO/IMU와 동기화 기준)
  "frame_id": <int>,             // 프레임 일련번호
  "windows": [                   // 검출된 창문 리스트 (0개 이상)
    {
      "order_index": <int>,      // 통과 순서 (색에서 매핑). §2.4 테이블 기준
      "color": <string>,         // (선택) "red" 등. order_index와 병기 여부는 §4.3
      "corners": [[u,v],[u,v],[u,v],[u,v]],  // 네 모서리 픽셀 좌표, 고정 순서
      "center": [u, v],          // 중심점 (편의)
      "det_conf": <float 0~1>,   // 탐지 신뢰도
      "color_conf": <float 0~1>  // 색 판정 신뢰도 (분리 제공 → 하류가 애매한 관측 필터링)
      // "depth": <float>        // §4.2 결정 시에만 추가
    }
  ]
}
```

### 4.2 depth 취급 [미정]

- **A. 픽셀 좌표만** (권장 기본): 이미지 처리기는 corner 픽셀만 제공, 3D 복원은 VIO/융합단이 pose 변화 기반 삼각측량으로.
- **B. depth 포함**: 시뮬레이터는 depth 이미지를 공짜로 주므로 시뮬에선 가능. 단, **하드웨어 이전 시 안 통하는 치트**임을 유의. 시뮬 전용 편의로만 사용.

### 4.3 순서 정보 표현 [미정]

- 정수 order_index만 / 색 문자열만 / **둘 다 병기** 중 택. (하류 사용 편의상 order_index 필수, color는 디버깅용 병기 권장)

### 4.4 카메라 파라미터 공유 [확정]

- intrinsics(fx, fy, cx, cy, 왜곡)와 기준 해상도는 **매 프레임 전송하지 않고 초기 1회 공유**.
- §2.1의 데이터 생성 카메라 값과 **반드시 동일**.
