# 창문 탐지 데이터셋 · 인터페이스 규격 (v0.2 — 확정본)

> 목적: 이미지 처리기(창문 탐지)의 **데이터셋 생성 규격**과 **VIO 전달 규격**을 하나의 문서로 통일한다.
> 이 문서 한 벌이 (1) 조윤호의 Isaac Sim 데이터 생성, (2) 류길남의 검출 모델 학습, (3) VIO 파트로의 데이터 전달을 모두 묶는 기준이 된다.
> v0.1 → v0.2: 미결 항목 전원 확정. 잔여 확인 사항은 §7 참조.

---

## 0. 문제 세팅 요약

- 창문(사각 개구부)들이 공간에 **임의 위치**로 흩뿌려져 있다.
- 통과 **순서는 색깔로 라벨링**되어 미리 정해진다.
- 이번 프로젝트의 실질 목표는 **시뮬레이션(Isaac Sim)** 구동. 하드웨어는 여유 시 확장.
- 데이터셋은 **Isaac Sim 내에서 자동 생성**(조윤호 담당)한다.

---

## 1. 이미지 처리기의 목표

카메라 프레임에서 **"다음에 통과할 창문이 지금 화면 어디에 있는가"**를 실시간으로 산출한다.

책임 범위:
1. 프레임 내 창문(사각 개구부) 검출 — **4-corner keypoint 방식 (YOLO-pose 기반)**
2. 창문 색 판정(**HSV 규칙 기반 후처리**) → 통과 순서 식별
3. 창문의 이미지상 기하 정보(네 모서리·중심) 추출

책임 밖 (VIO·융합단의 몫): 창문의 3D 위치 복원, 드론 자기 위치·자세 추정.

## 2. 카메라 · 좌표 규격 [확정]

| 항목 | 값 | 비고 |
|---|---|---|
| 원본 해상도 | **1280 x 720 (720p)** | 데이터 생성·좌표 기준 해상도 |
| 모델 입력 해상도 | 별도 (예: 640, 학습 시 확정) | 내부 리사이즈용. 외부 좌표와 무관 |
| **좌표 기준** | **모든 corner/center 픽셀 좌표는 원본 720p 기준** | 리사이즈 후 좌표를 밖으로 내보내지 않는다 |
| 좌표계 | 좌상단 원점 (u→오른쪽, v→아래) | 이미지 표준 |
| 컬러 포맷 | RGB | |
| intrinsics | fx, fy, cx, cy, 왜곡계수 | 데이터 생성 카메라 값을 §6에 기록, **초기 1회 공유** (매 프레임 전송 X) |

## 3. 색(=순서) 정의 [확정]

**3색으로 시작, 가변 확장 가능하게 설계한다.**
색↔순서 매핑은 코드에 하드코딩하지 않고 **설정 파일(config)로 분리**한다. 색 추가 시 테이블만 갱신.

### 3.1 색↔순서 매핑 테이블 (초기값)

| color | order_index | HSV 판정 구간 (H, OpenCV 0~179 기준) |
|---|---|---|
| red | 0 | H ∈ [0,10] ∪ [170,179], S ≥ 100, V ≥ 80 |
| green | 1 | H ∈ [50,70], S ≥ 100, V ≥ 80 |
| blue | 2 | H ∈ [110,130], S ≥ 100, V ≥ 80 |

- 시뮬 창문 색(윤호)과 판정 코드(류길남)는 **이 테이블을 단일 기준**으로 삼는다.
- 판정 구간은 시뮬 렌더 색 확인 후 미세조정 가능 (테이블만 수정, 코드 불변).
- 채도 높은 원색 재질 사용 권장 (조명 랜덤화에 대한 강건성 확보).

## 4. 데이터셋 생성 규격 (Isaac Sim / 조윤호)

### 4.1 배치 랜덤화 (도메인 랜덤화)

| 항목 | 범위 | 비고 |
|---|---|---|
| 창문 개수(씬당) | 1~5개 | 한 프레임 다중 등장 허용 |
| 위치 | 카메라 전방 작업 공간 내 랜덤 | |
| 거리 | 근·중·원거리 균등 | 원거리 소형 창문 학습 |
| 각도(yaw/pitch) | 정면~비스듬(±60° 내외) | 기울어진 창문 corner 학습 |
| 조명 | 밝기·방향 랜덤 | 색 판정 강건성 |
| 색 | §3.1 테이블의 3색, 색별 균등 분포 | |

### 4.2 부분 가림/잘림 라벨 정책 [확정: A로 시작 → C로 확장, 포맷은 처음부터 C 대응]

- **1차 데이터셋 (파이프라인 검증·초기 학습용)**: 정책 A — 네 corner가 모두 화면 안에 완전히 보이는 창문만 라벨. 빠른 제작 우선.
- **2차 데이터셋 (모델 동작 확인 후)**: 정책 C — corner별 **visibility 플래그** 도입. 화면 밖/가림 corner도 추정 좌표 + visibility=0으로 라벨.
- **라벨 포맷은 처음부터 visibility 필드를 포함**한다 (1차에서는 전부 1로 채움). 2차 전환 시 포맷 변경 없음.
- 근거: 드론이 창문 통과 직전에는 corner가 화면 밖으로 나가는 게 정상 상황. A만으로는 접근 최종 구간 탐지가 학습되지 않으므로 C 확장은 필수 경로다.

### 4.3 라벨 포맷 [확정: YOLO-pose txt]

이미지당 txt 1개, 창문(객체) 1개당 1행:

```
<class> <cx> <cy> <w> <h> <u1> <v1> <vis1> <u2> <v2> <vis2> <u3> <v3> <vis3> <u4> <v4> <vis4>
```

- class: order_index (red=0, green=1, blue=2) — §3.1 테이블
- cx, cy, w, h: 바운딩 박스 (YOLO 관례에 따라 0~1 정규화)
- (u_i, v_i, vis_i): corner 좌표(정규화) + visibility (0 또는 1)
- **corner 순서 고정: 좌상 → 우상 → 우하 → 좌하** (창문 정면 기준, 시계방향)
- 저장 구조: `images/{train,val,test}/*.png` + `labels/{train,val,test}/*.txt`
- 분할: train 80% / val 10% / test 10%
- 규모: 1차 3,000장 내외 → 2차 확장

### 4.4 GT 스트림 (파이프라인 검증용)

모델 학습 완료 전, 전체 파이프라인 검증에는 시뮬레이터 ground-truth corner를 §5 메시지 형식 그대로 흘려보낸다. 즉 **GT 스트림도 §5 규격을 따른다** — 모델 교체 시 하류 수정 없음.

## 5. VIO 전달 규격 (프레임 단위 출력) [확정]

### 5.1 메시지 구조

```json
{
  "timestamp": 1234567890123456789,   // int, ns. 카메라 프레임 취득 시각 (VIO/IMU 동기화 기준)
  "frame_id": 42,                     // int, 프레임 일련번호
  "windows": [                        // 검출된 창문 리스트 (0개 이상)
    {
      "order_index": 0,               // int. §3.1 테이블 기준 (필수)
      "color": "red",                 // string. 디버깅용 병기
      "corners": [[u1,v1],[u2,v2],[u3,v3],[u4,v4]],  // 720p 기준 픽셀. 순서: 좌상→우상→우하→좌하
      "corner_vis": [1,1,1,1],        // corner별 visibility
      "center": [u,v],                // 중심점 (편의)
      "det_conf": 0.97,               // 탐지 신뢰도 0~1
      "color_conf": 0.97              // 색 판정 신뢰도 0~1 (분리 제공 → 하류 필터링용)
    }
  ]
}
```

### 5.2 확정 사항

- **depth 미포함**: 이미지 처리기는 픽셀 좌표만 제공. 3D 복원은 VIO/융합단이 pose 변화 기반 삼각측량으로 수행. (시뮬 depth는 하드웨어 이전 시 통하지 않는 치트이므로 규격에서 배제)
- **순서 표현**: order_index(int) 필수 + color(string) 디버깅용 병기.
- **좌표 기준**: 항상 원본 720p 픽셀 좌표 (§2).
- **intrinsics**: 매 프레임 전송하지 않음. 초기 1회 공유 (§6).
